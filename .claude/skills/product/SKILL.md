# Product Skill

## When to Use

- Writing or reviewing Product Requirements Documents (PRDs)
- Defining user stories and acceptance criteria
- Prioritizing features or roadmapping
- Analyzing user feedback or market requirements
- Conducting competitive analysis
- Defining success metrics (KPIs, OKRs)

## Guidelines

1. **Start with user problems**, not solutions. Clearly articulate the pain point before proposing features.
2. **Define success metrics** for every feature. If you can't measure it, reconsider the requirement.
3. **Write atomic user stories** in the format: "As a [user type], I want [goal], so that [benefit]."
4. **Include acceptance criteria** using Gherkin syntax (Given/When/Then) when applicable.
5. **Consider edge cases and constraints**: technical limitations, regulatory requirements, accessibility.
6. **Prioritize using ICE or RICE frameworks** when helping with backlog prioritization.
7. **Always question scope creep**: ask "Does this solve the core problem?" before adding features.

## PRD Structure

When asked to write a PRD, follow this structure:

1. **Problem Statement** — What user pain point are we solving?
2. **Goals & Success Metrics** — How do we know we succeeded?
3. **User Stories** — Who are the users and what do they need?
4. **Acceptance Criteria** — What are the specific requirements?
5. **Out of Scope** — What is explicitly not included?
6. **Open Questions** — What needs further investigation?
7. **Timeline / Milestones** — Key dates and deliverables.
