# YEZZ 上线前阻断问题清单（Launch Blockers）

> **原则：不修复这些问题，上线不如不上线。**
> 以下 16 个 Issue 全部解决前，产品不应面向真实用户发布。
> **最后更新：** 2026-06-17

---

## 验收标准

上线前必须通过以下全部检查项：

- [ ] 所有占位图已替换为真实内容图片
- [ ] 联系页无原始 i18n key 暴露
- [ ] API fallback 数据已移除或替换为真实数据
- [ ] 中文页面标签/时长显示为中文
- [ ] 购物车空字段有明确的错误提示
- [ ] Hero CTA 跳转目标正确
- [ ] Hero 按钮文字与实际跳转一致
- [ ] 派对页有明确的预约/咨询出口
- [ ] 购物车 Session 绑定请求来源，无法跨用户操作
- [ ] Swagger UI 在生产环境不可访问
- [ ] 文件上传增加 MIME magic bytes 校验
- [ ] API 不返回任何密码字段（明文或哈希）
- [ ] 生产环境图片正常加载（需确认是配置还是代码问题）
- [ ] Redis 宕机时服务仍有限流保护
- [ ] 取消预约后对应时段容量正确回滚
- [ ] 预约提交时校验时段 ID 和剩余容量

## 阻断问题清单（16 项）

### 🔐 安全与合规（5 项）

#### SEC-005：Swagger UI 公开暴露

| 字段 | 内容 |
|------|------|
| **类别** | 安全 |
| **严重程度** | 阻断上线 |
| **位置** | `plugins/swagger.ts` |
| **问题** | Swagger UI 在所有环境无条件注册，攻击者可获取全量 API 路由结构 |
| **为什么必须修** | 等于把家门钥匙挂在大门口。攻击者拿到接口文档后可直接针对性攻击。 |
| **修复方向** | 增加环境变量开关：`if (process.env.NODE_ENV === "production" || process.env.ENABLE_SWAGGER === "false") return;` |
| **预估工作量** | 10 分钟 |

#### SEC-001：登录无暴力破解保护

| 字段 | 内容 |
|------|------|
| **类别** | 安全 |
| **严重程度** | 阻断上线 |
| **位置** | `routes/v1/auth.routes.ts` |
| **问题** | `/api/v1/auth/login` 无任何 Rate Limit，可被暴力破解 |
| **为什么必须修** | Admin 账号一旦被爆破，所有数据暴露，且用户侧数据也可能被篡改。 |
| **修复方向** | 复用现有 `checkRateLimit`，限制同一 IP 5 次/15 分钟 |
| **预估工作量** | 30 分钟 |

#### SEC-006：Cart Session IDOR

| 字段 | 内容 |
|------|------|
| **类别** | 安全 |
| **严重程度** | 阻断上线 |
| **位置** | `routes/v1/cart.routes.ts` |
| **问题** | Cart 只校验 session UUID，不验证请求者是否有权操作该 session |
| **为什么必须修** | 用户 A 知道用户 B 的 session UUID 即可修改/清空 B 的购物车，GDPR/PIPL 数据隐私风险。 |
| **修复方向** | **最小改动：** 在 `cart_sessions` 增加 `ipHash` 字段，创建时写入 `sha256(request.ip)`，后续操作时校验匹配。<br>**更完善：** session ID 存 httpOnly + secure + sameSite=lax cookie，服务端校验 cookie。 |
| **预估工作量** | 2-4 小时（取决于方案） |

#### SEC-003：新建用户密码明文返回

| 字段 | 内容 |
|------|------|
| **类别** | 安全 |
| **严重程度** | 阻断上线 |
| **位置** | `services/admin/users.admin.service.ts` |
| **问题** | `createAdminUser` 和 `resetAdminUserPassword` 响应中包含 `initialPassword` / `newPassword` 明文 |
| **为什么必须修** | 密码在 HTTP 响应中裸奔，即使只返回一次也是合规硬伤，且可能被日志/代理记录。 |
| **修复方向** | 响应中 `omit(passwordHash)`，密码仅通过邮件发送，API 响应中完全不返回密码字段。 |
| **预估工作量** | 30 分钟 |

#### SEC-004：文件上传 MIME 可绕过

| 字段 | 内容 |
|------|------|
| **类别** | 安全 |
| **严重程度** | 阻断上线 |
| **位置** | `services/admin/upload.admin.service.ts` |
| **问题** | 只检查 `file-type` 的 MIME，可被伪造扩展名绕过 |
| **为什么必须修** | 可上传恶意文件（如伪装成图片的脚本），如果配合其他漏洞可导致 XSS 或服务器沦陷。 |
| **修复方向** | 增加 magic bytes 校验 + 文件大小上限 + 扩展名白名单（仅允许 jpg/png/webp/pdf 等） |
| **预估工作量** | 2-3 小时 |

### 🏗️ 基础设施（2 项）

#### BUG-002：生产图片全部无法加载

| 字段 | 内容 |
|------|------|
| **类别** | 基础设施 |
| **严重程度** | 阻断上线 |
| **位置** | 待确认（可能是 `fly.toml` / 环境变量 / 存储配置） |
| **问题** | 所有上传的图片在生产环境显示为破图 |
| **为什么必须修** | 网站等于没有图片，对于依赖视觉展示的项目型业务是致命问题。 |
| **修复方向** | **先诊断：** 检查生产环境的 `MINIO_ENDPOINT` / `S3_BUCKET` / CDN 域名配置是否正确。<br>**如果是配置：** 修正环境变量即可。<br>**如果是代码：** 检查图片 URL 拼接逻辑。 |

#### SEC-002：Redis 宕机时限流失效

| 字段 | 内容 |
|------|------|
| **类别** | 基础设施 |
| **严重程度** | 阻断上线 |
| **位置** | `lib/cache.ts` 限流逻辑 |
| **问题** | Redis 不可用时，限流策略可能完全失效 |
| **为什么必须修** | 一旦 Redis 出问题或网络抖动，服务可被瞬间打挂（DDoS 或爬虫刷爆）。 |
| **修复方向** | 确认 `lib/cache.ts` 的 Redis 降级策略是否覆盖了限流路径。如未覆盖，限流失败时应拒绝请求（fail-closed）而非放行。 |
| **预估工作量** | 1-2 小时 |

### 🎨 品牌信任崩塌（5 项）

#### CON-001：所有图片是随机占位图

| 字段 | 内容 |
|------|------|
| **类别** | 品牌 |
| **严重程度** | 阻断上线 |
| **位置** | 全站图片 |
| **问题** | 项目图片、关于页图片等使用随机占位图（如风景/人物网图） |

| **修复方向** | 替换为真实项目照片、场地照片、团队照片。如暂无真实照片，宁可先隐藏图片区域或使用纯色背景+文字，也不要用随机图。 |

#### BUG-005：API 故障时显示假数据

| 字段 | 内容 |
|------|------|
| **类别** | 品牌 |
| **严重程度** | 阻断上线 |
| **位置** | 前端 fallback 数据（如联系信息、地址） |
| **问题** | API 请求失败时，前端显示写死的假地址、假电话 |
| **为什么必须修** | 用户可能真的按假地址导航或拨打假电话，造成实际损失和品牌灾难。 |

### 💼 核心业务逻辑（2 项）

#### BIZ-001：取消预约不归还容量

| 字段 | 内容 |
|------|------|
| **类别** | 业务逻辑 |
| **严重程度** | 阻断上线 |
| **位置** | `services/admin/bookings.admin.service.ts` |
| **问题** | Admin 取消预约后，time_slot 的 booked_count 没有回滚 |
| **为什么必须修** | 时段实际有空位但系统显示"已满"，新用户无法预约，直接流失。即使还没真实用户，demo/测试时也会暴露这个问题。 |
| **修复方向** | 在 `updateStatus` 中增加逻辑：当状态从非 cancelled 变为 cancelled 时，执行 `incrementBookedCount(timeSlotId, -numberOfPeople)` |
| **预估工作量** | 1 小时 |

#### BIZ-002：无时段 ID 跳过容量检查

| 字段 | 内容 |
|------|------|
| **类别** | 业务逻辑 |
| **严重程度** | 阻断上线 |
| **位置** | `services/bookings.service.ts` |
| **问题** | `activityType !== 'experience'` 时不校验 time_slot 容量，可超售 |
| **为什么必须修** | 预约系统的核心约束就是容量。如果某些类型可以跳过容量检查，系统无法保证不超售，品牌信誉受损。 |
| **修复方向** | 所有需要预约的类型都必须校验 time_slot 的剩余容量，`activityType` 不能成为跳过校验的理由。 |
| **预估工作量** | 1-2 小时 |

## 明确不在本清单中的事项（这些不是上线 blocker）

以下事项属于**优化、工程体验、运营效率**范畴，可以上线后分批处理：

- ❌ 前端数据层迁移到 TanStack Query + Zustand（当前 useEffect 能工作，只是样板代码多）
- ❌ Admin 列表分页（数据量小的时候不卡）
- ❌ 路由请求体验证 schema（当前 service 层有手工校验，功能正常）
- ❌ 错误监控 / 结构化日志 / 请求追踪 ID（运营侧问题，用户无感知）
- ❌ 健康检查检查下游依赖（运维监控问题）
- ❌ 邮件品牌包装 / 404 页面 / SEO 优化（体验杀手，但不是阻断）
- ❌ 数据库索引 / Session 清理 / 状态机约束（运营拖累）
- ❌ 购物车可删除 / 加购反馈 toast（体验优化）
