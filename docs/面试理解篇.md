## ① 并发、竞态、事务、锁 —— 这是你的"皇冠明珠",必须烂熟

这是面试官最爱挖的一块,也是最能体现"你真懂后端"的地方。把这一节彻底搞懂,你就赢了一半。

### 1.1 先建立直觉:什么叫"竞态条件 (race condition)"

**生活类比**:一个会议室,容量只剩 1 个空位。小明和小红几乎同时打开订座系统:
- 小明看到"剩 1 个" → 心想"我能订"
- 小红也看到"剩 1 个" → 心想"我也能订"
- 两人都点了确认 → **系统订出去 2 个,超了**

问题不在任何一个人,而在于:**两个人"读余量"和"写扣减"这两步交叉(interleave)了**。两人都在"还没人扣减"的瞬间读到了旧的余量。

> 一句话定义:**当结果取决于多个并发操作的执行先后顺序时,就有竞态。** 程序里"先读后写"且读写之间能被别人插队,就是典型竞态。

### 1.2 你的 double-booking bug,用时间线看清楚

你的预约场景有个字段 `bookedCount`(已订人数),余量 = `capacity - bookedCount`。

**取消预约**本来要做两件事:
1. 把预约状态改成 `cancelled`
2. 把容量加回去:`bookedCount -= 人数`

最初这两步是**分开的、没在一个事务里**。看并发出问题的时间线(A 在取消,B 在抢订同一时段):

```
时间 →
A: 改状态为 cancelled  ──┐
                         │  (此刻容量还没加回去,但状态已经变了)
B:        读到余量 = 0 ──┘  → B 以为满了,被拒  ❌ 本该能订
A: 容量 +回去 (bookedCount -1)
```

或者反过来的交叉,会让 `bookedCount` 被算错,出现"明明取消了却还占着位"或"两个人占同一个位"。**根因:状态和容量两个写操作之间存在一个"不一致的时间窗",别的请求在这个窗口里读到了错误的中间状态。**

### 1.3 修复武器一:事务 (Transaction) —— 保证"要么全做,要么全不做"

**事务 = 把多个操作打包成一个不可分割的整体(原子性)。** 数据库保证:这一包操作对外要么全部生效、要么全部不生效,中间状态别人看不到。

**生活类比**:银行转账,"A 扣 100" 和 "B 加 100" 必须在一个事务里。否则扣了还没加的瞬间崩了,钱就凭空消失。

你的修复:把"改状态"和"恢复容量"放进**同一个 `db.transaction`**:

```ts
await db.transaction(async (tx) => {
  await repo.updateStatus(id, "cancelled", tx);     // 操作1
  await slotsRepo.incrementBookedCount(slotId, -people, tx); // 操作2
}); // 两步一起提交,中间窗口消失
```

> 注意:这两个操作都传了同一个 `tx`(事务句柄),它们才在同一个事务里。这是面试官可能问的细节。

代码位置:`apps/api/src/services/admin/bookings.admin.service.ts:149-168`

**但是 ⚠️**:光有事务还不够防"两个人同时抢"。事务保证"我这一包的原子性",但不阻止"另一个事务同时也在读同一行旧值"。所以下单方向还需要**锁**。

### 1.4 修复武器二:行锁 `SELECT ... FOR UPDATE`(悲观锁)

回到会议室例子。要根治超卖,得让"读余量 → 判断 → 扣减"这一整段对同一个时段**串行执行**(一个处理完另一个才能开始)。

`SELECT ... FOR UPDATE` 的作用:**读这一行的同时给它上锁,别的事务想读/改这一行必须排队等我提交。**

**生活类比**:进会议室订位前,先把"这个时段的登记本"锁在自己手里。你查余量、扣减、合上本子(提交事务)之前,别人碰不到这本子,只能在门外等。等他拿到时,看到的已经是你更新后的余量。超卖不可能发生。

你的下单逻辑:

```ts
await db.transaction(async (tx) => {
  const slot = await slotsRepo.findByIdForUpdate(timeSlotId, tx); // 🔒 锁住这一行
  if (slot.capacity - slot.bookedCount < people) throw "已满";    // 在锁内判断
  await slotsRepo.incrementBookedCount(slot.id, people, tx);      // 在锁内扣减
}); // 提交,释放锁,下一个排队的人才进来
```

代码:`apps/api/src/services/bookings.service.ts:82-103` + `repositories/time-slots.repository.ts:134`(`.for("update")`)

### 1.5 一个容易被追问的细节:为什么扣减用 `bookedCount = bookedCount + delta`(SQL 表达式)而不是先读再写?

因为"先在 JS 里读出 `bookedCount`,加一,再写回去"本身就是个读-改-写竞态。直接写成 SQL `SET bookedCount = bookedCount + 1`,让**数据库在一条语句内原子完成**,就没有中间窗口。代码:`time-slots.repository.ts:122-132`。

### 1.6 悲观锁 vs 乐观锁(必背的对比)

| | 悲观锁 (你用的) | 乐观锁 |
|---|---|---|
| 思路 | "我假设一定会冲突",先上锁再操作 | "我假设很少冲突",不上锁,提交时检查 |
| 实现 | `SELECT ... FOR UPDATE` | 加 `version` 字段,更新时 `WHERE version = 旧值`,没改到就重试 |
| 适合 | **写争抢激烈、必须绝不出错**(订座、扣库存) | 冲突罕见、读多写少(改资料) |
| 代价 | 锁会让并发请求排队,吞吐略低 | 冲突时要重试,冲突多了反而更慢 |

**你的选择理由(背下来)**:预约/库存是"多个请求抢同一行 + 业务要求绝不超卖",**正确性优先**,悲观锁逻辑直观、不用写重试循环,所以选它。

### 1.7 Redis 在闪购里到底干嘛(别答错)

很多人会答"库存放 Redis 扣减"。你的安全答法:
- **Redis 扛"读"**:热点商品详情/库存的读走缓存,减轻数据库压力 + 配合限流削峰。
- **库存正确性(写)由 DB 行锁兜底,不靠缓存。** 因为缓存没有事务和锁的强一致保证。
- **客户端防抖 ≠ 防超卖**:防抖只是去掉重复点击、改善体验,真正的闸门永远在数据库。(这是陷阱题,记住"防抖不是安全手段"。)

代码佐证(yezz 同构):缓存 `apps/api/src/lib/cache.ts:35-79`,限流 `cache.ts:81-104`。

## ② 永远不信任客户端 (Never trust the client) —— 一句话覆盖两条 bullet

这是后端安全的**第一原则**。理解了它,"价格服务端校验"和"前后端双权限"就是同一个道理。

**为什么**:浏览器里的一切(表单值、JS 变量、发出的请求体)用户都能改。F12 改个价格、Postman 伪造个请求,轻而易举。所以**凡是涉及钱、权限、数据完整性的判断,前端的结论一律不算数,后端必须重新验一遍**。前端做校验只是为了**体验**(即时反馈、少跑一趟网络),不是为了安全。

### 应用 A:价格服务端权威 (server-authoritative pricing)

**问题**:前端展示了 ¥68 的商品,如果直接信前端传来的价格下单,用户把请求里的 68 改成 1,就 1 块钱买走了。

**做法**:下单时后端拿 `projectId` **回查数据库里的当前真实价格**,以 DB 价为准,忽略前端传来的 price。

> 心法:**前端的价格是"显示用的",后端从数据库查出来的价格才是"收钱用的"。**

代码:`apps/api/src/services/cart-orders.service.ts:97-117`(用 `projectId` 回查、用 DB 的 `priceRange` 覆盖客户端 price)。

### 应用 B:前后端双层权限

**问题**:前端把"管理员菜单"藏起来了,是不是就安全了?**不是。** 藏起来只是不显示,用户照样能直接用 Postman 打那个 admin API。

**做法**:两层各司其职:
- **前端(体验层)**:`AdminShell` 挂载时查登录态,没登录就跳登录页;按角色过滤菜单;staff 访问 admin 页就重定向。目的是"别让用户看到点不动的东西"。
- **后端(安全层)**:每个 admin 路由挂 `requireAdmin` 钩子,校验 JWT + 断言 `role === 'admin'`,否则 403。**这才是真正的闸门**,绕过前端也进不来。

> 心法:**前端权限是装修,后端权限是承重墙。** 装修可以拆,承重墙不能动。

代码:前端 `apps/web/components/admin/AdminShell.tsx:55-75`;后端 `apps/api/src/plugins/auth.ts:64-70`。

> 面试金句:"前端做权限是为了体验,后端做权限是为了安全;因为客户端不可信,所以安全判断必须在服务端。"

## ③ 状态机 (State Machine) —— 为什么不用一堆 if

**问题**:预约有 `new / contacted / confirmed / cancelled` 四态。如果不管理,会出现非法跃迁:已取消的又被改成确认、已确认的跳回 new……数据就乱了。

**状态机做法**:用一张表显式声明"每个状态**允许**变成哪些状态",其余一律拒绝。

```ts
const VALID_TRANSITIONS = {
  new:       ["contacted", "confirmed", "cancelled"],
  contacted: ["confirmed", "cancelled"],
  confirmed: ["cancelled"],   // 确认后只能取消,不能退回
  cancelled: [],              // 终态,哪都去不了
};
```

**为什么比散落的 if 好**:
1. **规则集中在一处**,一眼看全所有合法路径,改规则只改这张表。
2. **可单测**:给定 from/to 就能断言放行还是拒绝。
3. **防脏数据**:非法跃迁在入口就被 `INVALID_TRANSITION` 拦掉,不会写进库。

**生活类比**:红绿灯。绿→黄→红→绿 是合法环;绿不能直接跳红。状态机就是把"哪些跳转合法"写死成规则。

代码:`apps/api/src/services/admin/bookings.admin.service.ts:42-68`。

## ④ 副作用旁路 —— 为什么邮件/埋点要放在事务外、还故意"吞掉"异常

**问题**:状态改成 confirmed 后要发邮件通知客户。如果把"发邮件"也放进数据库事务里,会怎样?
- 发邮件依赖第三方(Resend),**慢**(几百毫秒甚至超时)又**可能失败**。
- 如果它在事务里且失败了 → **整个事务回滚 → 客户的预约状态没改成功**。这荒谬:邮件没发出去,凭什么把已经成功的业务也撤销?

**做法**:
1. 数据库事务**先提交**(业务数据落定)。
2. **事务之外**再发邮件,用 `try/catch` 把异常**吞掉只记日志**。

> 心法:**核心业务数据的一致性 > 通知/埋点的可达性。** 邮件、短信、埋点都是"旁路副作用",失败了不能拖垮主流程。

**生活类比**:你网购下单成功(主流程),结果"订单确认短信"没发出来 —— 你的订单当然还在,不会因为短信失败就被取消。

代码:`apps/api/src/services/admin/bookings.admin.service.ts:174-213`(事务在 149-168 已提交,邮件在后面,异常被 catch)。
> 同理埋点上报也是"失败不阻塞主流程"的旁路设计。

## ⑤ 统一响应信封 + 集中错误处理 —— 为什么值得做

**问题(没有信封时)**:每个接口返回格式五花八门,前端每个调用点都要自己判断 HTTP 状态码、解析错误、写一堆重复的 try/catch。错误处理散落各处,容易漏、不一致。

**信封做法**:后端**所有**响应统一成两种形状之一:
- 成功:`{ success: true, data: ... }`
- 失败:`{ success: false, error: { code, message } }`

前端在**一个地方**(`parseResponse`)统一拆信封:`success` 为 false 就抛一个带 `code`/`status` 的 `AdminApiError`,为 true 就直接返回 `data`。

**带来的好处(因果链)**:
- 业务调用点拿到的**永远是干净的 data**,不用再判状态码 → 代码简洁。
- 所有错误走**同一条路径**(抛 `AdminApiError`),可以在一处统一弹 toast、统一处理 401 跳登录 → 一致、不漏。
- 后端这边由全局 error handler 把 `AppError` 自动序列化成失败信封,业务代码只管 `throw new AppError(...)`,不用每个路由手写错误响应。

> 心法:**把"重复的、容易出错的横切逻辑"收口到一个地方做。** 这是 DRY 和"单一出口"思想。

代码:前端 `apps/web/lib/admin/api.ts:25-51`;后端 `apps/api/src/lib/response.ts` + `plugins/error-handler.ts`。

## ⑥ 轮询 vs 实时推送 —— 为什么选 30s 轮询(这是"工程权衡"题)

**需求**:后台导航上显示未读预约/订单的红点徽章,要"比较实时"。

**三种方案的本质区别**:
| 方案 | 怎么工作 | 代价 |
|---|---|---|
| 短轮询 (你用的) | 前端每 30s 主动问一次后端"有几条未读" | 有空轮询开销 + 最多 30s 延迟 |
| SSE | 服务端单向持续推送给前端 | 要维护长连接 |
| WebSocket | 双向长连接,服务端有事就推 | 长连接 + 断线重连 + 推送基础设施,最复杂 |

**你的选择理由(背下来)**:后台是**低频内部工具**,同时在线就几个员工,徽章晚 30s 显示完全可接受。WebSocket 要维护长连接、心跳、断线重连一整套基础设施,**投入产出不划算**。所以选最简单够用的轮询。**如果未来要做到秒级、或并发量上来,我会升级成 SSE。**

> 心法:**别上来就用最炫的技术,选"满足需求的最简方案"。** 面试官想听的是你会权衡,而不是会堆技术。

**一个加分细节**:定时器必须在组件卸载时 `clearInterval` 清理,否则会内存泄漏 + 卸载后 setState 报错。代码:`apps/web/components/admin/AdminShell.tsx:77-89`(`useEffect` 的 cleanup 返回 `clearInterval`)。

## ⑦ 分层架构 —— 为什么要 routes / services / repositories 三层

**问题(不分层时)**:把 SQL 查询、业务逻辑、HTTP 参数处理全塞在一个路由函数里 → 没法复用、没法单测、改一处牵动全身。

**三层各管一件事**:
- **routes**:只管 HTTP —— 收参数、校验、挂鉴权钩子、调 service。
- **services**:业务逻辑 —— 事务、状态机、规则判断。**不关心 HTTP,所以能脱离请求单测。**
- **repositories**:只管数据库 —— 用 Drizzle 查/写。**可被多个 service 复用**(如 `time-slots.repository` 同时被"下单"和"取消"用)。

> 心法:**每层只有一个变化的理由(单一职责)。** 换了数据库只改 repository,改了接口形状只改 route,业务规则只在 service。

**生活类比**:餐厅。服务员(route)只负责点单传菜,厨师(service)负责按菜谱做菜,采购(repository)只负责从仓库拿食材。厨师不用管客人坐哪桌。

代码:`apps/api/src/{routes,services,repositories}/`。
