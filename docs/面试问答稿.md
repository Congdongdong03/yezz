### Bullet 1 ✅ — 端到端交付 10+ 后台模块(表格 / 搜索 / 分页 / CRUD,Fastify REST + PostgreSQL/Drizzle)

**可能被问:**
- "10+ 模块具体是哪些?分层怎么组织的?"
- "一个典型请求从前端到数据库经过哪几层?"
- "为什么用 Drizzle 而不是 Prisma / 原生 SQL?"

**参考答案:**
> 后台覆盖预约、订单、档期、项目、派对套餐、画廊、分类、站点设置、用户、看板共 10 个模块,前台公开站再加预约/购物车下单。我没有只做某一层,而是每个模块从 DB schema → 仓储层 → service → 路由 → 前端页面整条做完。
>
> 分层很清晰:`routes`(参数校验 + 鉴权钩子)→ `services`(业务逻辑、事务、状态机)→ `repositories`(只负责 Drizzle 查询)。这样业务逻辑可以脱离 HTTP 单测,仓储层可以被多个 service 复用(比如 `time-slots.repository` 同时被下单和取消用到)。
>
> 选 Drizzle 是因为它是"贴近 SQL 的类型安全",schema 即 TS 类型,`$inferSelect` 直接生成行类型,不像 Prisma 有额外的生成步骤和运行时;团队改字段时类型立刻报错,重构很安全。

**代码佐证:**
- 仓储层目录:`apps/api/src/repositories/*.repository.ts`(11 个)
- service 层:`apps/api/src/services/` 与 `services/admin/`
- 三层串联示例:`routes/v1/admin/bookings.routes.ts` → `services/admin/bookings.admin.service.ts` → `repositories/bookings.repository.ts`
- 行类型推导:`bookings.admin.service.ts:70` `type BookingRow = typeof bookings.$inferSelect`

### Bullet 2 ✅ — 预约状态机 + 原子事务恢复容量 + 状态变更自动邮件(把恢复挪进事务修复了 double-booking)

> 这是最值钱的一条,面试官一定会深挖。务必能口述整条因果链。

**可能被问:**
- "什么是状态机?为什么需要它?"
- "double-booking 那个 bug 具体怎么发生的?你怎么定位、怎么修的?"
- "为什么一定要放在一个事务里?不放会怎样?"
- "下单时怎么防止超卖?用了行锁吗?为什么是 `FOR UPDATE` 而不是乐观锁?"
- "邮件为什么不在事务里?邮件发失败了订单会回滚吗?"

**参考答案(STAR):**
> **状态机**:预约有 `new → contacted → confirmed → cancelled` 四态,我用一张 `VALID_TRANSITIONS` 表显式定义合法跃迁,非法跃迁(比如已取消又改确认)直接抛 `INVALID_TRANSITION`。好处是状态流转的规则集中在一处、可单测,不会散落在各 if 里。
>
> **Bug 还原**:取消预约时要把对应时段的已订人数加回去(恢复容量)。最初"改状态"和"恢复容量"是两步分开写的。在并发场景 —— 一个请求在取消、另一个在重订同一时段 —— 两步之间存在时间窗,导致容量被错误计算,出现 double-booking / 超卖。
>
> **修复**:把"更新状态"和"恢复容量"包进**同一个 `db.transaction`**,要么全成功要么全回滚,消除了中间不一致窗口。同时只有从非取消态变为取消态、且有 timeSlot 和人数时才回补,避免重复回补。
>
> **下单方向**:创建预约时用 `findByIdForUpdate`(`SELECT ... FOR UPDATE` 行锁)锁住该时段行,在锁内校验剩余容量再 `bookedCount += people`。选悲观行锁是因为预约写入有真实并发争抢同一行、且业务要求"绝不超卖"(正确性优先),悲观锁逻辑直观、不需要重试循环。
>
> **邮件**:发邮件依赖第三方(Resend),慢且会失败,**故意放在事务提交之后**,并用 try/catch 吞掉异常只记日志 —— 邮件发失败绝不能让已成功的预约回滚。这是有意的权衡:业务数据一致性 > 通知可达性。

**代码佐证:**
- 状态机:`services/admin/bookings.admin.service.ts:42`(`VALID_TRANSITIONS`)、`:59`(`validateStatusTransition`)
- 事务内恢复容量:`bookings.admin.service.ts:149-168`
- 下单行锁防超卖:`services/bookings.service.ts:82-103` + 仓储 `repositories/time-slots.repository.ts:134`(`.for("update")`)
- 原子自增:`time-slots.repository.ts:122`(`bookedCount = bookedCount + delta`,用 SQL 表达式避免读-改-写)
- 邮件在事务外、异常被吞:`bookings.admin.service.ts:174-213`

### Bullet 3 ✅ — 角色感知权限,Next.js admin shell 与 Fastify `requireAdmin` 双端一致

**可能被问:**
- "前端做了权限,为什么后端还要做一遍?"
- "shell 是怎么在挂载时解析登录态的?未登录怎么处理?"
- "admin 和 staff 区别是什么?staff 访问 admin-only 页面会怎样?"
- "前端隐藏了菜单是不是就安全了?"

**参考答案:**
> 核心原则:**前端权限只是体验,后端权限才是安全边界**。两端各做一遍但职责不同。
>
> 前端(AdminShell):挂载时调 `getMe()` 解析当前用户;失败就 `router.replace('/admin/login')`;根据 `role` 过滤导航(`adminOnly` 项 staff 看不到);并且对 staff 直接访问 admin-only URL 的情况做兜底重定向到 `/admin/bookings`(`STAFF_BLOCKED_PREFIXES`)。这是为了 UX,不让用户看到点不动的菜单。
>
> 后端(Fastify):每个 admin 路由挂 `requireAdmin` 钩子,它先 `authenticate`(校验 JWT,优先 Bearer 头、其次 httpOnly cookie),再断言 `role === 'admin'`,否则 403。**即使有人绕过前端直接打 API,也进不来。** 鉴权信息来源同一个 JWT,所以两端对"谁是 admin"的判断一致。
>
> 所以"隐藏菜单"绝不等于安全,真正的闸门在 `requireAdmin`。

**代码佐证:**
- 前端解析登录态 + 未登录重定向:`components/admin/AdminShell.tsx:55-68`
- staff 兜底重定向:`AdminShell.tsx:33-40` + `:70-75`
- 按角色过滤导航:`AdminShell.tsx:103`
- 后端钩子:`apps/api/src/plugins/auth.ts:64-70`(`requireAdmin`)、`:43-62`(`authenticate`,Bearer + cookie 双通道)

### Bullet 4 ✅ — typed API client(统一响应信封 + 集中错误处理)+ 30s 轮询导航徽章

**可能被问:**
- "统一响应信封长什么样?好处是什么?"
- "集中错误处理是怎么做的?每个调用点还要写 try/catch 吗?"
- "30s 轮询为什么不用 WebSocket / SSE?轮询的缺点?组件卸载了定时器怎么办?"

**参考答案:**
> **统一信封**:后端所有响应统一为 `{ success: true, data }` 或 `{ success: false, error: { code, message } }`。前端 `parseResponse` 在一处判断 `success`,失败就抛带 `code`/`status` 的 `AdminApiError`,成功就直接返回 `data`。这样**业务调用点拿到的永远是干净的 data 或一个统一异常**,不用每处都解信封、判状态码。后端这边由全局 error handler 把 `AppError` 统一序列化成这个信封,业务代码只管 `throw new AppError(...)`。
>
> **轮询选型**:后台要在导航上显示未读预约/订单的红点徽章。我用 30s `setInterval` 轮询 `unread-count`。没上 WebSocket 是务实权衡 —— 后台是低频内部工具、并发用户就几个员工,徽章延迟 30s 完全可接受,而 WS 要维护长连接、断线重连、服务端推送基础设施,投入产出不划算。轮询缺点是有空轮询开销和最长 30s 延迟,如果未来要做到秒级或量大了,我会换成 SSE。
>
> **内存泄漏防护**:定时器在 `useEffect` 的 cleanup 里 `clearInterval`,路由切换/卸载时清掉,避免重复定时器和卸载后 setState。

**代码佐证:**
- 信封类型 + 集中解析/抛错:`apps/web/lib/admin/api.ts:25-51`(`parseResponse`、`AdminApiError`)
- 后端信封:`apps/api/src/lib/response.ts` + 全局错误处理 `plugins/error-handler.ts:5-13`
- 30s 轮询 + cleanup:`components/admin/AdminShell.tsx:77-89`

### Bullet 6/7 ⚠️ — 餐厅浏览/结算 + 全栈购物车(前端 Hooks,后端定价 subtotal/运费/折扣,下单时校验"所见价=所付价")

### Bullet 8 ⚠️ — 闪购防超卖:Redis 缓存 + DB 事务 + 行锁 + 客户端防抖

**可能被问:**
- "超卖是怎么发生的?你用什么锁?悲观锁 vs 乐观锁怎么选?"
- "Redis 在这里扮演什么角色?库存放 Redis 还是 DB?"
- "客户端防抖能防超卖吗?(陷阱题:不能,只是减轻)"

**参考答案:**
> 超卖的根因是**并发请求读到了相同的剩余库存**然后都去扣减。我的修法是 DB 事务 + 行锁:扣库存前 `SELECT ... FOR UPDATE` 锁住库存行,在锁内判断余量再扣减,把并发串行化到那一行上,保证不超卖(正确性优先,所以选悲观锁而不是乐观重试)。Redis 用于扛读 —— 热点商品详情/库存读走缓存减轻 DB 压力,并配合限流削峰;但**最终库存正确性由 DB 行锁兜底,不靠缓存**。客户端防抖(去重重复点击)只是减少无效请求、改善体验,**它绝不能作为防超卖手段**,真正的闸门在数据库。

**代码佐证(yezz 同构实现,可现场展示):**
- 行锁 + 事务内扣减:`apps/api/src/services/bookings.service.ts:82-103`、`repositories/time-slots.repository.ts:134`(`.for("update")`)
- Redis 缓存读 + 写失效:`apps/api/src/lib/cache.ts:35-79`
- Redis 限流(incr + expire,带内存兜底):`cache.ts:81-104`

### Bullet 13 ⚠️ — ECharts 商家分析看板(基于 SQL 聚合:日订单、营收趋势、商品表现)

> ⚠️ yezz **没有** ECharts,这条要靠你自己讲清楚,重点准备。

**可能被问:** "数据量大时聚合查询慢怎么办?图表数据实时还是预聚合?ECharts 大数据量渲染卡顿怎么处理?"

**参考答案要点(自备):**
> 看板按日/周维度展示订单数、营收趋势、商品销量 Top。数据来自 SQL 的 `GROUP BY` 聚合(按天 + 商品维度)。性能上:对高频查询的趋势数据可以预聚合到汇总表 / 定时物化,避免每次扫全表;给时间和商品维度建索引。ECharts 端数据点过多时用区域缩放(dataZoom)、按需降采样,避免一次渲染上万点卡顿。

### Bullet 14 ⚠️ — 阿里云 OSS 流式图片上传 + 异步状态反馈(改流式修复大文件超时)

> ⚠️ yezz 用的是 S3/MinIO/R2 不是阿里云 OSS,但**对象存储 + 服务端上传的概念高度同构**,可借力讲;"流式修超时"要能讲清原理。

**可能被问:** "为什么流式能修超时?之前怎么超时的?前端怎么拿到上传进度/异步状态?直传 OSS 还是经后端中转?"

**参考答案要点:**
> 最初是把整个大文件读进内存再上传,大图时内存占用高 + 请求超时。改成**流式上传**(边读边传,不把整文件载入内存)后,内存平稳、超时消失。前端通过异步状态反馈(上传中/成功/失败)给用户即时反馈,而不是卡住等。
> yezz 里的对象存储抽象可作同构佐证:`apps/api/src/lib/storage.ts`(S3 client,`forcePathStyle` 兼容 MinIO/R2)、上传服务 `services/admin/upload.admin.service.ts`、路由 `routes/v1/admin/upload.routes.ts`。
> 准备:能说清"流式 = 不缓冲整文件",以及阿里云 OSS SDK 的 stream/分片上传大致形态。

## 三、面试官常见"通杀"追问(提前备好)

| 追问 | 一句话答法 |
|---|---|
| "你这站流量大吗?凭什么说 high-traffic?" | 诚实:这是 DIY studio 预约站,真实流量不高。我做的是**并发正确性**防护(并发取消/重订不超卖),不是大流量压测。行锁+事务的代码是真的,可现场看。 |
| "悲观锁 vs 乐观锁怎么选?" | 写入争抢同一行、要求绝不超卖 → 悲观行锁,逻辑直观无需重试;读多写少/冲突罕见 → 乐观锁(version 字段)。我这里选悲观。 |
